# Review: openspec-ext-hack-through-test

**Date:** 2026-03-17
**Reviewer perspective:** Executing agent (Claude Code / Codex-class agent)
**Scope:** All files in the skill directory — SKILL.md, all references, script, agents config

---

## Overall Assessment

The skill is well-structured with a clear three-mode design, good separation between durable design guidance (propose/revise) and operational execution (run), and thoughtful guardrails. The reference documents are high-quality — the principles doc and checklist in particular give concrete, actionable criteria rather than vague guidance.

The main friction points are in **run mode**, where the gap between "what to do conceptually" and "how to do it mechanically" leaves room for execution mistakes by a literal-minded agent.

---

## What Is Clear

1. **Mode selection logic** — the decision tree in "Mode Selection" is unambiguous for most cases, with a sensible "ask if wrong mode would waste substantial work" escape hatch.

2. **Propose mode** — the most self-contained subskill. Goal, output shape, workflow steps, and the checklist reference all align. An agent can execute this without ambiguity.

3. **Revise mode** — similarly well-grounded. The checklist maps cleanly to artifact types (proposal, design, specs, tasks), and the instruction to use OpenSpec CLI before assuming layout is clear.

4. **Shared OpenSpec Context Rules** — the CLI commands, the fields to extract (`changeDir`, `schema`, `contextFiles`), and the "do not assume files exist" guardrail are all actionable.

5. **Principles doc** — the "translation rule" (temporary tactic -> permanent capability) is the single most useful heuristic in the skill, and it's stated clearly.

6. **Propose-revise checklist** — excellent. The "Smells" section at the bottom is particularly useful as a negative-space check.

7. **Script** — robust, well-structured, handles edge cases (gitignore, normalization, conflict detection). Output format (`KEY=VALUE`) is parseable.

8. **Guardrails** — all six are clear and prevent the most obvious mode-crossing mistakes.

## What Is Unclear or Likely to Cause Execution Errors

### 1. Script path resolution in run mode (HIGH)

**SKILL.md line 165-166:**
```bash
bash ./scripts/create_snapshot_worktree.sh --topic TOPIC_SLUG
```

The `./scripts/` path is relative — but relative to what? The SKILL.md preamble says "Use only the files bundled inside this skill directory," which implies the agent should know the skill directory path. But most agents run from the repo root, not from the skill directory. The agent must resolve the full path to the script before invoking it.

**What I'd do as the executing agent:** Resolve the skill directory path (e.g., from the magic-context submodule path) and construct the absolute path. But I'd be guessing at the resolution strategy.

**Suggestion:** Either (a) state the path resolution convention explicitly, e.g., "The skill directory is `magic-context/skills/openspec-ext/openspec-ext-hack-through-test/` relative to the repository root," or (b) show the invocation with a placeholder like `bash <skill-dir>/scripts/create_snapshot_worktree.sh`.

### 2. Topic slug derivation (MEDIUM)

The script requires `--topic TOPIC_SLUG` but the SKILL.md doesn't say what the slug should be. Should it be derived from:
- The OpenSpec change name?
- A user-provided label?
- A timestamp?
- The change name sanitized to hyphen-case?

The script has `normalize_topic_slug` that lowercases and replaces non-alphanumeric runs with hyphens, so passing the raw change name would *work*, but the SKILL.md doesn't state this intent.

**Suggestion:** Add a line like "Use the change name as the topic slug; the script normalizes it to hyphen-case."

### 3. "Start logs using the bundled templates" is underspecified (MEDIUM)

**SKILL.md line 175:** "Start logs using the bundled templates."

Concretely, what does this mean? The agent needs to:
1. Parse the `KEY=VALUE` output from the snapshot script
2. Copy the log template to `LOG_ROOT/session-log.md` (or some filename — which?)
3. Fill all the `<...>` placeholders with values from the script output and the OpenSpec context

None of these steps are stated. The template itself is wrapped in `````markdown ... ````` fences (a code block within a code block), so the agent must also understand to extract the inner content, not copy the fences.

**Suggestion:** Add concrete steps, e.g.:
- "Parse the script's `KEY=VALUE` output lines."
- "Create the session log at `<LOG_ROOT>/<timestamp>-session.md` using `references/log-template.md`."
- "Fill the header placeholders from the script output and the OpenSpec context gathered earlier."

### 4. What "execute the next step in the canonical path" means concretely (MEDIUM)

**SKILL.md line 177:** The hack-through-testing loop says "execute the next step in the canonical path" but doesn't specify:
- Should the agent `cd` into the worktree first?
- Should commands run against the worktree or the original repo?
- What if the canonical path is a multi-step sequence — how does the agent decompose it?

The `run-mode-openspec-adaptation.md` helps ("one CLI command, one integration test command, one build-then-run sequence, one multi-step agent-driven interaction sequence") but the SKILL.md itself doesn't bridge from "inspect the change to find the command" to "here's how to run it in the worktree."

**Suggestion:** Add a note like "Execute all commands inside the throwaway worktree, not the original checkout. Use `cd <WORKTREE>` or pass `--repo <WORKTREE>` to commands that accept it."

### 5. Stop rules are undefined (MEDIUM)

**SKILL.md line 181:** "continue until success, stop-rule exhaustion, or a high-risk boundary"

The log template has a `**Stopping rule:**` field but there's no default stopping rule. Should the agent:
- Stop after N issues?
- Stop after a time budget?
- Stop after the first high-risk boundary?
- Define the stopping rule itself based on the change scope?

An agent without guidance here will either run indefinitely or stop too early.

**Suggestion:** Provide a default stopping rule, e.g., "Default stop after 5 unresolved issues, 60 minutes wall time, or the first blocker that requires unsafe external side effects."

### 6. Commit message format (LOW)

The issue template shows `hack-through: <issue-id> <short workaround>` as the commit message format, but this convention isn't mentioned in the SKILL.md workflow steps. An agent following only the SKILL.md would have to discover this in the issue template.

**Suggestion:** Mention the commit message convention in the run-mode workflow, e.g., "Commit verified workarounds with the message format: `hack-through: <issue-id> <short workaround>`."

### 7. Cleanup guidance is absent (LOW)

The log template has a "Cleanup" section asking whether the throwaway branch/worktree were kept or deleted. But neither the SKILL.md nor any reference says when or how to clean up. Should the agent:
- Always delete the worktree at session end?
- Leave it for the user to inspect?
- Ask the user?

**Suggestion:** Add a cleanup step at the end of the run-mode workflow, e.g., "After synthesis, ask the user whether to keep or remove the throwaway worktree. Record the decision in the session log."

### 8. OpenSpec CLI availability assumption (LOW)

The skill assumes `openspec` is on PATH and functional. If it isn't, the agent would hit errors in revise and run modes with no guidance on what to do. This is likely fine in practice (the hosting project has OpenSpec set up), but a preflight note would help.

### 9. Script output parsing convention (LOW)

The script prints `KEY=VALUE` lines. The SKILL.md never says "parse the script output." An agent that just runs the script and ignores stdout would miss all the paths needed for the log template.

**Suggestion:** In run-mode step 3, add: "Capture the script's stdout. Each line is a `KEY=VALUE` pair: `REPO_ROOT`, `SOURCE_HEAD`, `TOPIC_SLUG`, `HTT_BRANCH`, `SNAPSHOT_COMMIT`, `HTT_HOME`, `WORKTREE`, `LOG_ROOT`, `ISSUES_DIR`, `RUNS_ROOT`."

### 10. Run artifact capture mechanics (LOW)

**SKILL.md line 178:** "record failures and save artifacts" and the log template says "Copy generated artifacts for each run to `<runs-root>/<run-ts>/`." But what counts as a "run artifact"? Stdout/stderr of the command? Generated files? The SKILL.md doesn't define this. The `run-ts` convention (timestamp per run) is only in the log template, not in the workflow steps.

## Potential Edge Cases Not Addressed

1. **Submodules in the target repo** — the git-snapshot-plumbing doc notes "Nested Git repositories must be snapshotted separately" but the skill doesn't tell the agent what to do if submodules are present. Likely answer: ignore them and note it in the session log.

2. **Branch name collision** — the script exits with an error if the branch already exists. The SKILL.md doesn't tell the agent how to recover (append a timestamp? ask the user? delete the old branch?).

3. **Concurrent sessions** — two agents running `run` mode on the same change with the same topic slug would collide. No guidance on unique naming.

4. **`$openspec-propose` and `$openspec-new-change` notation** — in propose mode, step 5 says "suggest a follow-up such as `$openspec-propose`." The `$` prefix appears to be a skill invocation convention. If the executing agent doesn't know this convention, it might treat these as shell variables or literal strings.

## Minor Observations

- The `agents/openai.yaml` `default_prompt` is clear and correctly reflects the three-mode structure.
- The `policy.allow_implicit_invocation: false` is appropriate — this skill should be explicitly requested.
- The log and issue templates are thorough without being bloated. The issue template's "Fix Attempt N" pattern is well-designed for iterative progress.
- The guardrail "Do not present temporary workarounds discovered in `run` mode as the final fix" is critical and well-placed.
